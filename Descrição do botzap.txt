### 📁 1. Estrutura de Arquivos do Projeto

Seu projeto é um bot para WhatsApp construído em **Node.js** usando a biblioteca `whatsapp-web.js`. A estrutura atual é composta por:

*   **`bot_sticker.js`**: É o "coração" do projeto. Contém toda a lógica, comandos e conexões com o WhatsApp.
*   **`package.json` / `package-lock.json`**: Gerenciam as dependências do projeto (bibliotecas de terceiros como
 `whatsapp-web.js`, `qrcode-terminal`, `dotenv`, etc.) e definem o ponto de entrada principal (`bot_sticker.js`).
*   **`.env`**: Arquivo oculto de variáveis de ambiente. É aqui que você guarda configurações sensíveis e IDs de grupos 
(`GROUPS_WELCOME`, `GROUPS_LOG`, `GROUPS_STICKERS`, `PORT`, etc.) sem expô-las no código fonte.
*   **`Dockerfile`**: Contém as instruções para "conteinerizar" a aplicação. É ele que permite rodar o bot em servidores 
Linux ou em nuvem com todas as dependências de sistema necessárias.
*   **`mensagens_grupo.txt`**: Um arquivo de texto local usado como banco de dados simples (log) para salvar as mensagens 
dos grupos monitorados.
*   **`.wwebjs_auth/`**: Uma pasta gerada automaticamente que guarda os "cookies" da sua sessão do WhatsApp. É ela que impede
 que você tenha que ler o QR Code toda vez que o bot reinicia.

### ⚙️ 2. Funções e Processos do `bot_sticker.js`

O código está dividido em áreas bem organizadas. Aqui está o que cada parte faz:

#### A. Configurações Iniciais (`CONFIG`)
Define o nome do autor das figurinhas e puxa os IDs dos grupos autorizados diretamente do arquivo `.env`.
 Isso garante que o bot só reaja onde ele tem permissão.

#### B. Cliente do WhatsApp (`new Client(...)`)
Inicia o WhatsApp nos bastidores.
*   **`LocalAuth`**: Salva a sessão localmente na pasta `.wwebjs_auth/`.
*   **`puppeteer`**: O bot não usa uma API oficial do WhatsApp; ele abre um navegador Google Chrome invisível 
(Headless) e opera o WhatsApp Web de forma autônoma. Os argumentos (`--no-sandbox`, `--disable-gpu`, etc.) 
são otimizações para fazer esse navegador rodar em servidores (VPS) sem interface gráfica e consumindo menos memória.

#### C. Servidor de Monitoramento (Health Check)
Cria um servidor web simples (porta 5500 ou a definida no `.env`). Isso não tem a ver com o WhatsApp em si,
 mas é **fundamental para hospedagens em nuvem** (como Render ou Heroku). Essas plataformas exigem que a aplicação
  "escute" uma porta web; caso contrário, elas acham que o bot travou e o desligam.

#### D. Eventos de Conexão
*   `qr`: Se não houver sessão salva, gera um QR Code no terminal para você escanear.
*   `authenticated` / `ready`: Avisam quando o bot conectou com sucesso.

#### E. Processo Principal (`client.on('message')`)
Atua como um **Roteador** sempre que uma nova mensagem chega:
*   **Área 1 (Logs)**: Se a mensagem vier de um grupo autorizado (`GROUPS_LOG`), ele chama a função auxiliar
 `saveMessageToLog()` para escrever o texto, o nome da pessoa e o horário no arquivo `mensagens_grupo.txt`.
*   **Área 2 (Figurinhas)**: Se a mensagem for exatamente `@stk` e estiver em um grupo autorizado (`GROUPS_STICKERS`), 
ele chama a função `handleStickerCommand()`.

#### F. Gestão de Membros (`client.on('group_join')`)
Monitora os grupos de `GROUPS_WELCOME`. Quando alguém novo entra no grupo, o bot puxa as informações do usuário e do grupo
 e envia uma mensagem padronizada de boas-vindas mencionando o usuário (`@número`) e enviando o link de um formulário Google Forms.

#### G. Funções Auxiliares
*   **`saveMessageToLog(msg)`**: Pega os dados da mensagem, formata a data/hora e adiciona uma nova linha no arquivo `mensagens_grupo.txt`.
*   **`handleStickerCommand(msg)`**: Verifica se a mensagem atual ou a mensagem que está sendo respondida (`msg.hasQuotedMsg`)
 contém uma imagem. Se sim, faz o download (`downloadMedia()`) e a reenvia imediatamente como um sticker/figurinha (`sendMediaAsSticker: true`).

### 🌐 3. Como o Bot funciona sem estar aberto no terminal?

Para que o bot funcione 24 horas por dia sem que você precise manter uma janela preta do terminal aberta no seu PC, 
entram em cena duas tecnologias que você já configurou no projeto: **Docker** e **Hospedagem em Nuvem/VPS**.

Veja como a mágica acontece nos bastidores:

1. **O Navegador "Invisível" (Headless)**
   A biblioteca `whatsapp-web.js` usa o Puppeteer. O Puppeteer abre o Google Chrome no modo *headless* (sem cabeça).
    Isso significa que o Chrome roda perfeitamente na memória RAM, processando o site do WhatsApp Web, mas sem gerar
     nenhuma janela gráfica (UI). Como não há interface gráfica, não há necessidade de uma tela ou de um usuário olhando para ela.

2. **A "Conteinerização" (`Dockerfile`)**
   Você possui um arquivo `Dockerfile`. Quando você roda o projeto usando o Docker (ou faz upload para uma nuvem que usa Docker), 
   ele cria uma "máquina virtual enxuta".
   Observe as linhas do seu `Dockerfile`:
   ```dockerfile
   RUN apt-get update && apt-get install -y fonts-ipafont-gothic ... libxss1 dbus dbus-x11 xvfb ... chromium
   ```
   Isso instala o Linux, as fontes de texto para idiomas variados e as bibliotecas gráficas falsas (`xvfb`, `dbus-x11`)
    necessárias para "enganar" o Chrome, fazendo-o pensar que está em um computador normal com tela, quando na verdade está 
    dentro de um servidor escuro em um Data Center.
   Ao final do Dockerfile, o comando `CMD ["node", "bot_sticker.js"]` diz ao servidor para ligar o bot assim que o container 
   for iniciado.

3. **Background Services (Daemons / Serviços de Nuvem)**
   *   **Se você roda em um VPS Linux**: Você provavelmente o executa usando o Docker ou um gerenciador de processos como o **PM2**
    (`pm2 start bot_sticker.js`). O PM2 desvincula o bot do seu terminal e o coloca rodando como um serviço do sistema operacional 
    em "segundo plano" (background). Se fechar o terminal, o serviço continua lá, operando.
   *   **Se você roda no Render/Heroku/Railway**: Você faz o upload dos seus arquivos (ou conecta o GitHub). Os computadores deles 
   leem o seu `Dockerfile`, ligam o container e, graças ao seu Servidor HTTP (da linha `http.createServer`), eles mantêm a máquina 
   ligada ininterruptamente. 

**Resumo da Ópera**: O seu bot não precisa do terminal porque ele é um script de servidor rodando um navegador Chrome invisível,
 mantido "acordado" por uma infraestrutura em nuvem, e validando sua autenticação pelos arquivos guardados na pasta `.wwebjs_auth`.
